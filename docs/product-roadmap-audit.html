Kresco Product Audit

Compact living checklist for turning Kresco into a cohesive Bac learning platform.

Updated: 2026-06-12 Strategy: targeted rewrites Priority: coherence before more features

Core Thesis

Product risk

Feature sprawl. The app needs shared models, admin workflows, reporting, and reliable states.

Main rewrites

Bac Exam Video Bank, Lesson Exercise Bank, payment/access, leaderboard/XP/badges.

Most missing layer

Trust/support: reports, access clarity, broken-content visibility, production monitoring.

Current Build Strategy

Feature-first phase

Build as many real product surfaces as possible now. Prioritize working workflows over perfect architecture.

Later hardening phase

After the app is feature-rich, audit security, performance, data model quality, UX states, tests, and dead code.

Minimum guardrail

Even during fast building, keep features isolated, clearly named, and easy to delete or rewrite.

Feature-First Guardrails

GuardrailReason
Use feature flags or hidden routes for rough features.Prevents unfinished work from leaking into student-facing flows.
Do not mix prototype data with production data.Protects future migration, AI data quality, billing, and analytics.
Keep experimental code grouped by feature.Makes later debloating possible without archeology.
Prefer vertical slices over giant half-built systems.Each feature should have a usable UI, API path, and data shape, even if rough.
Write down known shortcuts.Hardening later only works if debt is visible.
Never fake payments, access, or student progress as real data.These become dangerous when users arrive.

Filters

Roadmap

Bac Exam Video Bank

critical

End-of-year Bac prep: exam questions, full exams, and video answers.

  • Browse by subject, year, session, topic, concept.
  • Video answer is primary; written statement/correction is supporting.
  • Track watched, saved, attempted, weak Bac themes.

Lesson Exercise Bank

critical

Year-round practice: many exercises per lesson, mostly text/formula/diagram corrections.

  • Filter by lesson, topic, difficulty, concept, status.
  • Corrections support formulas, diagrams, hints, steps.
  • Video optional; not required for value.

Payment and Access Rewrite

critical

Students must always know what they own, what is locked, why, and how to unlock.

  • Owned subjects, tier, entitlements, payment status.
  • Support Pro/VIP/Platinum, packages, refunds, coupons, manual grants.
  • Payment/access issues route to support.

XP, Leaderboard, Badges

critical

Add learning actions first, then balance XP rigorously.

  • Actions: exercise attempts, corrections, notes, ratings, live, AI.
  • Caps, anti-farming, idempotency, seasons, leagues.
  • Badges with rules, rarity, inventory, profile display.

XP and Gamification Overhaul

critical

Current XP is too simple for rewards, rankings, anti-abuse, or learning quality.

  • Create a clear XP economy doc: what earns XP, how often, caps, reversals, and rationale.
  • Add weighting by subject, topic/course difficulty, exam relevance, and learning action value.
  • Wire or remove unused rewards: retry correct, exam complete, perfect quiz, daily login, streak bonus.
  • Add XP seasons: weekly/monthly/semester leaderboards, not only all-time ranking.
  • Add first-class XP reversal/adjustment transactions for bugs, abuse, or admin correction.
  • Add daily anti-farming caps by category: quiz-correct, lesson complete, retry, video, lab, exam.
  • Expand quests: subject quests, weak-area quests, streak quests, exam-prep quests, mistake-review quests.
  • Remove fake fallback XP/leaderboard data in production; show empty/error states instead.
  • Add mastery model: per-concept mastery, confidence, decay, spaced repetition, weak-topic detection.
  • Add XP audit dashboard: explain why a user has a given XP total and which awards/reversals created it.
  • Improve frontend gamification display so XP, quests, badges, leagues, and progress are understandable.

Mistake Notebook

critical

More valuable than generic notes: turns weak points into revision.

  • Wrong quizzes, failed exercises, rewatched corrections.
  • Repeated weak concepts and "retry later" queue.
  • Links back to exact question, correction, video timestamp.

Non-Technical Admin

critical

SQLAdmin is not enough for staff.

  • Content studio: topics, videos, exercises, quizzes, summaries.
  • Review professor suggestions; publish with versioning.
  • Finance, users, entitlements, support, analytics.

Backoffice and Finance

critical

Production needs a permissioned backoffice, not a generic admin panel plus finance tab.

  • /backoffice zones: users, content, live, finance, security.
  • /admin SQLAdmin becomes super-admin emergency tooling only.
  • Money history must be append-only: provider events, transactions, ledger entries, refunds, manual grants.
  • Paid access comes from entitlement records; users.is_pro is only a projection/cache.

Admin Needs Attention

critical

Admin home should show what is wrong or waiting.

  • Broken videos, failed payments, reports, failed uploads.
  • Missing corrections, unreviewed suggestions, live issues.
  • AI answers reported wrong, low-rated content.

Report Problem Everywhere

critical

Small report action on videos, exercises, corrections, quizzes, live, comments, AI, payment/access.

  • Types: broken video, wrong correction, typo, bad explanation.
  • Missing answer, payment/access issue, inappropriate comment, app bug.
  • Capture URL, user, content context, timestamp, device, screenshot if possible.

Help and Support Center

critical

Central support for account, payment, access, videos, exercises, live, chat.

  • Clear locked-content explanation everywhere.
  • Account/access status page.
  • Support escalation for unresolved cases.

Content Quality Status

high

Admin-visible quality state for every important content object.

  • Draft, reviewed, published, needs fix, deprecated.
  • Missing correction, missing video, broken media.
  • Separate quality from paid access gates.

Empty and Failure States

high

Dedicated audit before launch.

  • No exercises, saves, live sessions, notes, correction, access, results.
  • Every state says what happened and what to do next.
  • Failures offer retry, report, support, or fallback.

Professor Quiz Suggestions

high

Close-team internal tool: professor submits structured quiz drafts; admin approves.

  • Quiz type, prompt, options/input, answer, explanation, difficulty.
  • Link to course, topic, lesson video, item, tab.
  • Admin previews, validates, publishes/rejects.

Quiz Architecture and Integrity

critical

Quizzes need one official path for rendering, authoring, grading, versioning, and progress.

  • Consolidate duplicate quiz UIs into one official renderer.
  • Make QuestionSet/Question the primary authoring model later; JSON should not be source of truth forever.
  • Add quiz versioning/snapshots so published edits do not muddy old attempts.
  • Ensure every XP/progress quiz route uses backend grading only.
  • Keep frontend demo/practice quizzes separate from official progress quizzes.
  • Add admin/content validation so broken quiz JSON cannot be published.
  • Store correct_answer_json carefully because it exists for audit in DB.

Professor Panel

high

Improve chat, live dashboard, settings, optional analytics.

  • Chat: search, filters, attachments, moderation, unread states.
  • Live: OBS help, states, recording, notifications, clarity.
  • Always scoped to CourseOffering and audited.

Student Settings

high

Complete account, security, personalization, notifications.

  • Name, avatar, banner, niveau, filiere, email, password.
  • Delete/export account policy.
  • Dark mode later.

Notes, Comments, Ratings

high

Polished study and feedback system.

  • Notes deep-link to item/tab/video timestamp/exercise.
  • Comments need moderation and visibility rules.
  • Ratings feed admin quality queues.

Structured Course Pages

high

Beautiful course pages, not only interactive labs.

  • Definitions, properties, theorems, figures, examples.
  • Formula, warning, method, common-mistake blocks.
  • Staff-editable blocks where possible.

Summary Downloads

high

Rigorous resource system, not loose file links.

  • Attach to subject, topic, item, concepts, gates.
  • Track opens, downloads, saves, broken files.
  • Version and replace safely.

AI Tutor and RAG

high

Start with Kresco retrieval plus strict Bac system prompt; fine-tune later if data quality supports it.

  • Use current screen, topic, item, tab, quiz, exercise, video minute.
  • Answer from transcripts, corrections, summaries, exercises.
  • Suggest videos, exercises, summaries, labs.

Data and Transcript Pipeline

high

Needed before serious AI or fine-tuning.

  • Timestamped transcripts with subject/topic/concepts.
  • Normalized exercises, corrections, summaries, quizzes.
  • Track source quality and student helpfulness signals.

Production Monitoring

critical

Know when providers or app flows fail.

  • CMI, virement, CashPlus/AshPlus cash rails, VdoCipher, Ably, email, media, AI, uploads, outbox.
  • Request IDs, error IDs, user/session context, provider logs.
  • Graceful degraded states and "last updated" indicators.

Scale Guardrails

critical

Small details that matter when real users arrive.

  • Idempotency: payments, XP, quiz submits, progress, reports, uploads.
  • Audit logs, rate limits, soft deletes, content versioning.
  • Moderation queues for comments, chat, reports, AI answers.

Ops Backlog

high

Infrastructure for maintainability.

  • Background jobs: transcripts, embeddings, checks, fanout, rollups.
  • Data export/deletion policy.
  • View-as-student with audit, feature flags, separate staging/demo data.

Remaining Weakness Audit

This is the compact cross-cutting weakness list. The roadmap above says what to build; this table says what can break the product if we ignore it while adding features fast.

AreaRepo signalWeaknessNeeded direction
Content production workflow Content authoring docs rely on SQLAdmin, seeds, manual status fields, and review checklists. No real pipeline for 40 exercises per lesson, Bac videos, transcripts, diagrams, corrections, QA, and fixes. Build import templates, validation, quality states, review queues, publishing gates, and ownership per content object.
Design system debt Figma source exists, but many pages/components still use custom classes, hard-coded colors, and one-off cards/skeletons. Student, professor, admin, auth, Zed, quiz, and animated-course surfaces can drift visually and behaviorally. Create shared primitives, design tokens, app shells, card/list/form states, and a UI debt audit for hard-coded surfaces.
Search architecture Separate search exists for courses, exam bank, admin, professor chat, and Zed formulas. No unified search model, result taxonomy, ranking, or indexing plan across lessons, exercises, videos, transcripts, and AI. Define search domains, result types, filters, permissions, transcript indexing, and future vector/hybrid search boundaries.
Notification rules Notifications and Ably channels exist, but product preference/routing rules are not defined. Risk of spam, missed alerts, expensive polling, unclear priority, and no student/professor control over channels. Write a notification matrix: event, audience, channel, priority, frequency, digest behavior, preference, and fallback state.
Moderation and safety Chat/comments/live questions exist or are planned; report queues are mostly roadmap items. No complete moderation workflow for bad comments, live chat, AI answers, wrong corrections, abuse, or takedowns. Add report queues, hide/delete/restore actions, moderation states, block tools, admin audit logs, and escalation policy.
Support operations Support center, payment/access support, report-everywhere, and customer-success workspace are TODOs. Real user issues will arrive through random channels unless the app has tickets, owners, statuses, and escalation. Create support tickets for access, payment, broken content, bugs, refunds, and confusion; add admin assignment and metrics.
Data model cleanup Docs say Topic/TopicItem is target language, while old chapter/lesson compatibility still appears in routes/docs/code. Legacy naming plus saved-item uncertainty can create future bugs, confusing admin UX, and bad content imports. Run a cleanup pass: remove unused compatibility, define resource taxonomy, rewrite saved items, and version content models.
Mobile and PWA readiness Responsive components exist, but no clear PWA/offline/install/push/native strategy is settled. Bac students may be mobile-first; cheap-phone performance, video UX, downloads, and offline behavior are unknown. Decide responsive web vs PWA vs native later; meanwhile add mobile budgets, key viewport tests, offline rules, and push plan.
Accessibility and i18n French localization exists in places and many controls have aria labels, but no formal accessibility/i18n policy. Hard-coded French/English mix, math notation, keyboard flows, contrast, screen-reader behavior, and Arabic/RTL are not audited. Make French the copy source, inventory hard-coded strings, audit keyboard/contrast/math alt text, and design Arabic/RTL later.
Content rights and licensing Legal TODOs exist, but content ownership for videos, exercises, corrections, summaries, and professor submissions is not explicit. Unclear ownership can block publishing, reuse, takedowns, AI training/RAG usage, and professor-created quiz workflows. Add contributor terms, content license records, third-party source tracking, usage rights, takedown process, and AI data policy.
