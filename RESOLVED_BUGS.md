- [x] **[HIGH]** `frontend/app/(dashboard)/exam/[subjectId]/page.tsx` - Timer Reset Bug in Exam Mode. The `setInterval` for the countdown timer is inside a `useEffect` that depends on `handleSubmit`, which changes on every answer selection, causing the timer to artificially pause and delay.

- [x] **[CRITICAL]** `backend/app/routers/professor.py` - In `_conversation_last_sender_role`, the query selects **ALL** messages for the given `conversation_ids` and loads them into memory, just to pick the most recent one using a python dictionary (`if conversation_id not in roles`). For conversations with thousands of messages, this will cause massive memory spikes and database load. It should use `DISTINCT ON (conversation_id)` or a window function subquery.
- [x] **[HIGH]** `backend/app/routers/courses.py` - In `_grade_quiz_question` for `image_hotspot` questions, `safe_rx` and `safe_ry` are calculated using `region.get("rx") - cursor.get("radius")`. A malicious student can submit a negative `radius` (e.g., `-10000`), which inflates `safe_rx` and `safe_ry` to massive positive numbers. This makes the elliptical boundary check `(dx^2 / rx^2) + (dy^2 / ry^2) <= 1` trivially evaluate to `True`, allowing them to bypass and cheat on hotspot questions automatically.
- [x] **[HIGH]** `backend/app/routers/payments.py` - In `stripe_webhook`, when handling `charge.dispute.created`, if the customer ID cannot be resolved, the server throws an `HTTPException(status_code=502)`. This causes Stripe to not receive a 200 OK, meaning Stripe will continuously retry the webhook delivery, potentially leading to webhook blocking or rate limiting issues. It should log the error and return 200 OK.
- [x] **[MEDIUM]** `backend/app/routers/users.py` - In `verify_email`, if the user has a `professor` role, `require_professor_active_offering` is called before setting `is_email_verified = True`. However, `require_professor_active_offering` inherently throws an error if the professor does not have an active offering. A newly registered professor without an offering will be permanently unable to verify their email, leading to a catch-22 edge case.
- [x] **[MEDIUM]** `backend/app/routers/gamification.py` - In `get_subject_plan`, `LessonProgress.lesson_id.in_(all_lesson_ids)` is executed without checking if `all_lesson_ids` is empty. If a subject has no lessons, this triggers a query with an empty `IN` clause (`IN (NULL)`), which throws an SAWarning and degrades database performance unnecessarily. It should be wrapped in `if all_lesson_ids:`.
- [x] **[LOW]** `backend/app/routers/courses.py` - In `submit_tab_quiz`, there is an unnecessary explicit row-level lock on the `User` table (`select(User.id)...with_for_update()`). Since quiz attempts use an idempotency hash and catch `IntegrityError` to handle race conditions, locking the entire `User` row is an anti-pattern that creates a bottleneck and potential deadlocks across all user mutations.
